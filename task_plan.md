## 排期建议

### P0：先收敛可交付的核心闭环
目标：优先把“家庭建组 -> 冰箱管理 -> AI 出菜 -> 烹饪指引 -> 购物补货”打通成稳定主流程。

- 登录与家庭组主链路稳定化
  说明：保留现有 `auth` / `group` 能力，优先补齐登录态、建组、邀请、切组后的页面跳转与异常态。
- 冰箱管理与拍照入库可用化
  说明：先确保冰箱 CRUD、识别结果回填、保质期录入、过期提醒基础规则可用，即使识别准确率后续再优化。
- AI 食谱生成主流程闭环
  说明：优先保证“基于现有食材生成食谱”和首页推荐这两条路径返回稳定、字段一致、可直接进入烹饪页。
- 烹饪指引与计时器稳定性
  说明：继续围绕 `cooking-guide`、`cooking-steps`、`CookingTimer.vue` 做稳定性收口，减少页面语法、倒计时与状态同步问题。
- 购物清单基本闭环
  说明：从食谱缺料生成购物项、勾选采购状态、家庭共享这条链路应作为 P0 完成项。
- P0 联调与回归
  说明：P0 结束前至少完成 H5 一轮端到端回归，并补一次微信小程序关键流程冒烟。

### P1：补强协作、留存与运营能力
目标：在核心闭环稳定后，补齐家庭协作和用户留存相关能力。

- 家庭协作任务页与流程
  说明：后端已有 `task` 模块，但前端协作入口尚不完整，建议在这一阶段补齐任务分配、状态流转、家庭动态联动。
- 通知与家庭动态
  说明：后端已有 `notification`、`feed`，优先补可见入口、已读状态、关键提醒的前端承接。
- 个人偏好、收藏/历史、营养信息
  说明：这些能力能明显提升推荐质量与复用率，适合放在主链路稳定后的增强阶段。
- Admin 食谱管理完善
  说明：继续完善管理端食谱表单、审核与发布体验，支撑用户端内容供给。
- 基础质量建设
  说明：在这一阶段补关键接口契约、错误提示一致性、埋点/日志/简单监控，降低后续返工成本。

### P2：体验创新与扩展能力
目标：在核心业务稳定后再投入高成本、长链路或高不确定性的能力。

- 视频/音频识别生成食谱
  说明：需求价值高，但涉及媒体上传、转写/识别、AI 结构化提取与长耗时任务处理，建议放到 P2。
- 食谱分享与社区化能力
  说明：分享、外链、社区互动更偏增长功能，不应挤占核心闭环交付节奏。
- 统计报表与周报/月报
  说明：后端已有 `stats` 模块骨架，但数据口径、展示方式、留存价值需要在主流程跑稳后再细化。
- 更重的语音、多媒体和跨平台优化
  说明：包括更完整的语音控制、媒体上传兼容、真机体验打磨、部署与发布流程优化。
