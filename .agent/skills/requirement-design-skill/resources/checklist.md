# 需求设计文档评估检查清单

## 📋 使用说明

这份检查清单用于评估需求设计文档的质量。每个检查项都有对应的分值,总分100分。

---

## 1. 完整性检查 (40分)

### 1.1 项目概述 (10分)

- [ ] **项目背景** (3分)
  - 清楚说明项目起源
  - 明确要解决的问题
  - 有充分的背景信息

- [ ] **项目目标** (3分)
  - 目标具体明确
  - 目标可衡量
  - 有明确的成功标准

- [ ] **目标用户** (2分)
  - 用户画像清晰
  - 用户需求明确

- [ ] **核心价值** (2分)
  - 价值主张清晰
  - 与用户需求匹配

### 1.2 功能需求 (15分)

- [ ] **功能列表** (5分)
  - 功能分类合理
  - 优先级明确
  - 覆盖核心场景

- [ ] **用户故事** (5分)
  - 格式规范(作为/我想要/以便)
  - 有明确的验收标准
  - 覆盖主要用户场景

- [ ] **功能详细说明** (5分)
  - 输入输出明确
  - 处理逻辑清晰
  - 异常处理完整

### 1.3 非功能需求 (8分)

- [ ] **性能要求** (2分)
  - 有具体的性能指标
  - 指标可测量

- [ ] **安全要求** (2分)
  - 认证授权方案明确
  - 数据安全有保障

- [ ] **兼容性要求** (2分)
  - 明确支持的平台/设备
  - 有具体的版本要求

- [ ] **可用性要求** (2分)
  - 有可用性指标
  - 有容错和恢复方案

### 1.4 技术方案 (7分)

- [ ] **技术栈** (2分)
  - 技术选型明确
  - 有选型理由

- [ ] **系统架构** (2分)
  - 架构设计合理
  - 有架构图或清晰描述

- [ ] **数据模型** (2分)
  - 核心实体定义清楚
  - 实体关系明确

- [ ] **接口设计** (1分)
  - 关键接口有说明

---

## 2. 清晰度检查 (30分)

### 2.1 语言表达 (10分)

- [ ] **准确性** (4分)
  - 术语使用准确
  - 没有歧义表述
  - 技术描述正确

- [ ] **简洁性** (3分)
  - 表达简洁明了
  - 没有冗余内容
  - 重点突出

- [ ] **一致性** (3分)
  - 术语使用一致
  - 格式统一
  - 风格一致

### 2.2 结构组织 (10分)

- [ ] **逻辑性** (5分)
  - 章节组织合理
  - 内容层次清晰
  - 逻辑连贯

- [ ] **可读性** (5分)
  - 使用标题、列表等格式
  - 有适当的图表
  - 排版美观

### 2.3 验收标准 (10分)

- [ ] **可测试性** (5分)
  - 验收标准明确
  - 可以验证是否达成
  - 有具体的测试方法

- [ ] **可衡量性** (5分)
  - 有量化指标
  - 指标可测量
  - 有明确的阈值

---

## 3. 可行性检查 (30分)

### 3.1 技术可行性 (10分)

- [ ] **技术成熟度** (4分)
  - 使用成熟稳定的技术
  - 技术风险可控
  - 有技术储备

- [ ] **实现难度** (3分)
  - 技术难度合理
  - 没有不可逾越的技术障碍
  - 有可行的实现方案

- [ ] **技术风险** (3分)
  - 识别了主要技术风险
  - 有应对措施
  - 风险可控

### 3.2 时间可行性 (10分)

- [ ] **时间估算** (4分)
  - 时间估算合理
  - 考虑了各个阶段
  - 有缓冲时间

- [ ] **里程碑设置** (3分)
  - 里程碑明确
  - 阶段划分合理
  - 可以跟踪进度

- [ ] **优先级** (3分)
  - 功能优先级明确
  - 核心功能优先
  - 可以分阶段实施

### 3.3 资源可行性 (10分)

- [ ] **人力资源** (4分)
  - 团队配置合理
  - 技能匹配
  - 人员充足

- [ ] **成本预算** (3分)
  - 有成本估算
  - 预算合理
  - 在可接受范围内

- [ ] **外部依赖** (3分)
  - 识别了外部依赖
  - 依赖可获得
  - 有备选方案

---

## 📊 评分标准

### 总分计算

- **完整性得分** = 完整性检查得分 (满分40)
- **清晰度得分** = 清晰度检查得分 (满分30)
- **可行性得分** = 可行性检查得分 (满分30)
- **总分** = 完整性 + 清晰度 + 可行性 (满分100)

### 等级划分

| 总分范围 | 等级            | 评价                              |
| -------- | --------------- | --------------------------------- |
| 90-100   | ⭐⭐⭐⭐⭐ 优秀 | 需求文档质量很高,可以直接进入开发 |
| 75-89    | ⭐⭐⭐⭐ 良好   | 需求文档质量较好,稍作完善即可     |
| 60-74    | ⭐⭐⭐ 及格     | 需求文档基本可用,但需要较多改进   |
| 0-59     | ⭐⭐ 不及格     | 需求文档质量不足,需要大幅修改     |

## 🔍 常见问题和改进建议

### 问题1：功能描述过于模糊

**表现**：使用"优化"、"改进"、"提升"等模糊词汇  
**改进**：使用具体的、可衡量的描述,如"将页面加载时间从5秒降低到2秒"

### 问题2：缺少验收标准

**表现**：只描述功能,没有说明如何验证  
**改进**：为每个功能添加明确的验收标准,使用"当...时,系统应该..."的格式

### 问题3：非功能需求缺失

**表现**：只关注功能,忽略性能、安全等  
**改进**：补充性能指标、安全要求、兼容性要求等

### 问题4：技术方案不明确

**表现**：没有说明使用什么技术,或者只是罗列技术名称  
**改进**：说明技术选型理由,提供架构设计,说明关键技术点

### 问题5：时间计划不合理

**表现**：时间估算过于乐观或没有考虑风险  
**改进**：基于历史数据估算,预留20-30%的缓冲时间,设置明确的里程碑

### 问题6：风险评估不足

**表现**：没有识别风险或只是形式化地列举  
**改进**：识别技术风险、资源风险、外部风险,并提供具体的应对措施

## 📝 使用示例

### 示例1：评估一个功能描述

**原始描述**：

> "系统应该支持用户管理"

**评分**：清晰度 2/10 (过于模糊)

**改进后**：

> "系统应该支持用户管理功能,包括：
>
> 1. 用户注册：用户可以通过邮箱注册账号
> 2. 用户登录：支持邮箱+密码登录,登录失败3次后锁定账号15分钟
> 3. 密码重置：用户可以通过邮箱接收重置链接
> 4. 个人资料编辑：用户可以修改昵称、头像、个人简介
>
> 验收标准：
>
> - [ ] 用户可以成功注册并收到验证邮件
> - [ ] 用户可以使用正确的邮箱和密码登录
> - [ ] 输入错误密码3次后,账号被锁定15分钟
> - [ ] 用户可以通过邮件链接重置密码"

**评分**：清晰度 9/10 (描述具体,有验收标准)

## ✅ 检查清单总结

使用这份检查清单时：

1. 逐项检查,标记完成的项目
2. 记录每个维度的得分
3. 计算总分和等级
4. 针对未通过的项目提出改进建议
5. 优先改进高优先级的问题

**记住**：需求文档的质量直接影响项目的成功率,投入时间完善需求文档是值得的！
